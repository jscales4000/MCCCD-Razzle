# 2026-05-31 — Lessons Learned: The Dead-Feedback Root Cause

**Status: RESOLVED.** Processor→panel feedback works end-to-end, by name, with zero
hardcoded joins. Verified live on the real panels (lav-mute icon lit from an
injected `MicLavMuteFb`; NVX HDMI-sync dots showing real device data).

This document is the post-mortem for an issue that consumed **multiple sessions**
(~3 days of "connecting contracts") before it was actually understood. The goal is
that **no one ever burns days on this again.** Read the "Best Practices" and
"New/Corrected Rules" sections even if you skip the narrative.

---

## 1. The symptom (what we saw)

- Switching a source on the panel *appeared* to work — the marker moved.
- **No processor→panel feedback worked**: power state, mic mutes, source confirmation,
  the 8 video-sync dots — all dead.
- "Other button presses don't work" — because buttons whose visible result depends on
  feedback (power, mic mute) looked dead even when the command was sent.

The one thing that looked like it worked (source markers moving) was a **lie**: it was
an *optimistic local mirror* in `router.ts` updating the UI without any SIMPL round-trip.
**We actually had zero working real feedback.**

---

## 2. The actual root cause (one bug, everything else was downstream)

**The `.cce` contract had every signal's direction inverted.**

Crestron Contract Editor encodes signal direction by `attributeType`, and the mapping is:

| attributeType | Contract Editor column | Direction | Generated C# in `Main.g.cs` |
|---|---|---|---|
| **0** | **State** | **feedback** (processor → panel) | `void <Name>(MainBoolInputSigDelegate cb)` — a **setter** |
| **1** | **Event** | **command / press** (panel → processor) | `event EventHandler<UIEventArgs> <Name>` — an **event** |

Our contract had it **backwards**: every `*Fb`/feedback was authored as an **Event
(attributeType 1)**, and every command (`DisplayPower`, `Display1Source`…) as a **State
(attributeType 0)**. Contract Editor therefore generated:

- `SystemPowerFb` as a C# **`event`** → the processor had **no way to *set* it**. Feedback
  was structurally impossible to send.
- `DisplayPower` as a **setter** → a command the panel can't really drive.

So feedback was dead at the contract level, before a single line of service code ran.

### Where the inversion came from

Following the FRED doc **"Crestron .cce Contract File Generation Guide"**, which states
*"commands = attributeType 0, feedbacks = attributeType 1."* **That is backwards relative
to the actual tool.** Every contract generated by following that guide is inverted. (See
"New/Corrected Rules" — the FRED guide must be fixed.)

### The compounding sin: bypassing the contract with raw joins

A prior session, finding that the generated wrappers wouldn't let them *set* feedback
(because feedbacks were un-settable events), concluded *"the generated wrappers are
misaligned"* and **hand-rolled a raw-join dispatcher** (`PanelDispatcher` + `PanelJoins`,
87 constants) that poked `panel.SmartObjects[1].BooleanInput[N]` directly.

That write goes to the wrong sig — `BooleanInput[4]` is the *command* `VolumeUp`, not the
*feedback* `SystemPowerFb` (which lived on `BooleanOutput[4]`). So feedback was **still
dead**, and now we'd also **violated the cardinal rule: never hardcode joins.** Every
subsequent workaround was built on this broken foundation.

### Missteps *this* session (intellectual honesty matters)

- Initially theorized "the contract is inverted" (**correct!**), then **talked myself out
  of it** by trusting the FRED `.cce` guide — pivoted to a "non-canonical .cce / broken
  siblingId" theory (**a wrong turn**). The user pushing back ("are we hardcoding joins?")
  and the GroundTruth test put us back on track.
- Chased a red herring: the cse2j's two-section "shape" (`states` keyed by `[so][join]`,
  `events` keyed by `[name]`). Irrelevant — it's normal Contract Editor output.

**Meta-lesson: when a written guide and the actual tool disagree, the tool wins. Verify,
don't reason from possibly-wrong docs.**

## 4. How we deviated from the rules

1. **"NEVER use raw join numbers. Always use `CONTRACTSYMBOL.SignalName`."** (CH5 Extended
   Developer, NEVER-DO #2.) Violated by `PanelDispatcher`/`PanelJoins`. **This was the
   load-bearing deviation** — it hid the real bug and spawned every workaround.
2. **Claiming success without real-panel verification.** A prior handoff asserted "SO1
   booleans work fine — no regression" without re-testing; the user found *nothing*
   worked. (See `superpowers:verification-before-completion`.)
3. **Escalating architecture instead of questioning the premise.** Cap-at-10 → SO2 →
   sub-contract → multi-component: five structural iterations chasing a misdiagnosis.
   `superpowers:systematic-debugging` Phase 4.5 is explicit: 3+ failed structural fixes =
   the architecture/assumption is wrong, stop and question it.

## 5. What worked — how we actually fixed it

1. **Used the existing CWS debug tool** (`/cws/aa140/debug/`) to *inject feedback* and
   isolate the path — proving feedback was globally dead, not device-specific.
2. **Built a minimal 2-signal "GroundTruth" contract**, ran it through Contract Editor, and
   **read the generated `Main.g.cs`.** This *empirically* established the `attributeType`↔
   direction encoding (feedback-as-event with no setter for attrType 1; setter for
   attrType 0). **This single experiment is what cracked it.** It cost ~20 minutes and
   ended days of guessing.
3. **Regenerated a canonical contract** (`build_aa140_cce.py`, 64 paired signals) with the
   *correct* encoding — feedbacks on State (attrType 0), commands on Event (attrType 1),
   bidirectional `siblingId`, VideoSync folded into the single `Main` component.
4. **Rewired everything to the name-based `Contract`** — `_contract.AA140.<Fb>(sig => …)`
   for feedback, `_contract.AA140.<Cmd> += handler` for commands. **Deleted** `PanelJoins`
   + `PanelDispatcher`.
5. **Removed the optimistic mirror** and the SO2 split; made the debug `/signal` endpoint
   inject **by name** (reflection on `IMain`).
6. **Verified live**: injected `MicLavMuteFb` by name → lav-mute icon lit (nothing else
   drives it); NVX HDMI-sync dots show real device data. The "SO2 never delivers" saga
   resolved as a *side effect* of fixing the root cause.

## 6. New / corrected rules to add to the standards

1. **CORRECTED ENCODING (critical):** In a `.cce`, `attributeType 0` = **State** =
   **feedback (proc→panel)** = generates a name-based **setter**; `attributeType 1` =
   **Event** = **command (panel→proc)** = generates a C# **event**. Put `*Fb`/status names
   on the **State** side. **The FRED "Crestron .cce Contract File Generation Guide" has
   this reversed and MUST be corrected** — following it inverts every contract.
2. **NEVER bypass the generated `Contract` with raw `SmartObjects[n].XInput[join]` writes.**
   If the generated wrappers look wrong, **the `.cce` is wrong — fix the `.cce`.** The
   raw-join bypass hides the real bug and silently writes the wrong sig.
3. **Prove the contract direction with a 2-signal test BEFORE authoring the full contract.**
   Build a minimal command+feedback `.cce`, run Contract Editor, and confirm the generated
   `Main.g.cs` exposes feedback as a **setter** and command as an **event**. Only then scale
   to N signals. (Keep `contracts/GroundTruth.cce` + `build_groundtruth_cce.py` as the
   reusable harness.)
4. **An "optimistic mirror" is a code smell, not a pattern.** It masks a broken feedback
   path. If you find yourself mirroring a publish into a feedback store, the real feedback
   is broken — fix that instead, and never let the mirror disguise it.
5. **Default to ONE component / ONE SmartObject.** Don't split into SO2 / sub-contracts /
   multi-component without a *proven* reason. The SO2 split solved a non-problem.
6. **The "SmartObject-1 caps at ~10 boolean inputs" claim is FALSE.** Delete it from your
   mental model. It was raw-join writes hitting wrong sigs.
7. **Verify feedback by injecting it on a real panel.** The CWS debug tool's
   `POST /signal?name=<Fb>&type=bool|ushort&value=X` is the standard check. Don't claim
   feedback works without it.
8. **When 3+ structural changes fail, stop and question the premise** — don't escalate the
   architecture (systematic-debugging Phase 4.5).

## 7. Quick reference — the name-based pattern (do this)

```csharp
// Feedback (processor -> panel): name-based setter
_contract.AA140.SystemPowerFb((sig, m) => sig.BoolValue = true);
_contract.AA140.Display1SourceFb((sig, m) => sig.UShortValue = srcIndex);

// Command (panel -> processor): name-based event
_contract.AA140.DisplayPower   += (s, a) => PowerToggle(a.SigArgs.Sig.BoolValue);
_contract.AA140.Display1Source += (s, a) => Route(a.SigArgs.Sig.UShortValue, 1);
```

```bash
# Verify any feedback by name (no joins):
curl -k -s -X POST --data "" \
  "https://192.168.2.198/cws/aa140/debug/signal?name=SystemPowerFb&type=bool&value=1"
```

`.cce` authoring (per signal): a **State** entry `{name:"<Fb>", attributeType:0}` paired
bidirectionally by `siblingId` with an **Event** entry `{name:"<Cmd>", attributeType:1}`;
`dataType` matched (1=bool, 2=numeric, 3=string); **no join numbers** (auto-allocated at
Build). See `contracts/scripts/build_aa140_cce.py` for the generator.
